## 1. Ідея проєкту

**Дослідницьке питання:** Чи рівномірно та раціонально місто планує розподіл громадського транспорту — у часі (години доби), між типами транспорту та між районами?

Працюємо з **плановою пропозицією**: розклади (GTFS Static) та місткість транспортних засобів доступні для завантаження прямо зараз, без збору історії. Дві ключові метрики:

- **Плановий інтервал руху (headway)** — скільки хвилин між рейсами маршруту за розкладом у різні години. Малий інтервал = маршрут у пріоритеті.
- **Провізна здатність** = кількість ТЗ на маршруті × пасажиромісткість. Показує, скільки пасажирів місто може перевезти, і чи відповідає це населенню районів.

Бонус (якщо все йде за графіком): запускаємо збирач GPS-позицій на добу і для кількох маршрутів показуємо «план vs факт» — як reality check у Discussion.

**Особливості даних — виявлено при реальному завантаженні (вірити цьому списку, а не описам порталу):**

- GTFS покриває **лише Київпастранс** (трамвай/тролейбус/автобус). Маршрутки — тільки списком ТЗ (№3), без розкладу → H3 порівнюємо через vehicles, не через headway.
- `stop_times.txt` містить лише **контрольні точки**, не всі зупинки. Для headway це ок: перша точка (stop_sequence = 1) = відправлення з кінцевої.
- `stop_sequence` починається **з 1** (опис на порталі каже «з 10» — файл каже інакше).
- `route_type`: 0 = трамвай (стандарт GTFS); повний перелік перевірити через `count(routes, route_type)` — опис порталу (1/2/3) суперечить файлу.
- `calendar.txt`: окремі service_id для буднів і вихідних → для аналізу брати **будній** (monday = 1).
- `vehicles` Київпастрансу: `routeId` буває **списком через кому** ("14,47") → `separate_rows()`.
- `vehicles` маршруток: `capacity` — **текст** («не менше 21»), багато null → чистка (плюс до Data Preparation).
- Realtime — **бінарний protobuf**, а не JSON, як пише портал (~400 ТЗ у знімку). Збирати як `.pb` (`R/collect_realtime.R`), парсити в CSV (`scripts/parse_realtime.py`, потребує `pip install gtfs-realtime-bindings pandas`).
- Відомі баги GTFS з пропозицій громадян на порталі (дзеркальні напрямки рейсів — виправлено у 2026; відсутність частини тимчасових маршрутів) → згадати в Discussion як limitation.

## 3. Гіпотези та тести

Використовуємо **тільки методи з курсу** (лекції 6–10): t-тести, Shapiro–Wilk / QQ / Kolmogorov–Smirnov, chi-square (goodness-of-fit та незалежність), рангові тести (Mann–Whitney, Kruskal–Wallis), bootstrap.

- **H1 (пік vs непік).** Планові інтервали в години пік (7–10, 17–20) менші, ніж у непікові.
  Спершу діагностика нормальності по групах (Shapiro–Wilk + QQ-плот, Лекція 7). Нормально → двовибірковий t-тест (Лекція 6); ні → Mann–Whitney U (Лекція 9). Кажемо в Methodology, чому обрали саме той тест.
- **H2 (розподіл між слотами / типами транспорту).** Інтервали відрізняються між 4 часовими слотами та між автобусом/тролейбусом/трамваєм/маршруткою.
  Тест: Kruskal–Wallis (Лекція 9) + post-hoc попарні Mann–Whitney (`pairwise.wilcox.test()` з поправкою на множинні порівняння).
- **H3 (зв'язок «тип ↔ частота обслуговування»).** Категорія частоти маршруту (частий ≤10 хв / середній 10–20 / рідкісний >20 у пік) не залежить від типу транспорту (або від перевізника: комунальний vs приватний).
  Тест: chi-square тест незалежності на таблиці спряженості (Лекція 8; за малих очікуваних частот — поправка Єйтса).
- **H4 (розподіл між районами).** Розподіл провізної здатності по районах не пропорційний розподілу населення.
  Тест: chi-square goodness-of-fit (Лекція 8): спостережена здатність по районах vs очікувана пропорційно населенню. Стандартизовані залишки показують недо-/переобслужені райони. Головна візуалізація — choropleth-карта здатності на 1000 мешканців.
- **H5 (місток до теорії ймовірностей).** Чи розподілені інтервали руху експоненційно (тобто чи є прибуття пуассонівським потоком)?
  Тест: Kolmogorov–Smirnov (Лекція 7) або chi-square GoF для функції розподілу (Лекція 8).
- **Бонус 1 (bootstrap, Лекція 10).** Bootstrap-довірчий інтервал для різниці середніх інтервалів пік vs непік — гарно доповнює H1.
- **Бонус 2 (план vs факт).** Для 2–3 маршрутів з добового збору GPS: чи відповідають фактичні інтервали плановим (парний/двовибірковий t-тест) — чесно позначаємо як пілот на даних вихідного дня.

### Крок 4. Планові headway (людина B, 2–3 год — серце проєкту)

**Що:** для кожного маршруту × напрямку × часового слоту — середній плановий інтервал між рейсами.
**Як (логіка):** інтервал найчистіше рахувати на **першій зупинці маршруту** (`stop_sequence == 1`) — це моменти відправлення рейсів з кінцевої. Відсортувати відправлення, взяти різниці сусідніх — це і є headway.

```r
library(tidyverse); library(lubridate)
dep <- gtfs$stop_times |>
  filter(stop_sequence == 1) |>
  inner_join(gtfs$trips, by = "trip_id") |>
  filter(service_id == ОБРАНИЙ) |>
  mutate(dep_sec = as.numeric(hms(departure_time)))   # обережно: у GTFS буває 25:10:00 (після півночі)

headways <- dep |>
  arrange(route_id, direction_id, dep_sec) |>
  group_by(route_id, direction_id) |>
  mutate(headway_min = (dep_sec - lag(dep_sec)) / 60) |>
  filter(!is.na(headway_min), headway_min > 0, headway_min < 120) |>
  mutate(hour = dep_sec %/% 3600,
         time_slot = case_when(hour %in% 7:9   ~ "ранковий пік",
                               hour %in% 10:16 ~ "день",
                               hour %in% 17:19 ~ "вечірній пік",
                               TRUE            ~ "вечір/ніч")) |>
  group_by(route_id, direction_id, time_slot) |>
  summarise(sched_headway_min = mean(headway_min), n_departures = n())
```

Санітарна перевірка: у метро/тролейбусів у пік headway має бути ~3–10 хв. Якщо виходить 40 — щось не так із service_id або парсингом часу.
**Готово, коли:** таблиця route × direction × slot із правдоподібними інтервалами.

### Крок 9. Тести (людина C, 2–3 год)

**Що:** H1–H5 тільки методами з курсу, з діагностикою перед кожним тестом — це 5 балів «Interpretation».
**Як:**
- **H1 (пік vs непік):** спершу діагностика по двох групах — `shapiro.test()` + QQ-плот (`qqnorm`/`qqline`). Нормально → `t.test(headway ~ peak)`; скошено (очікувано для інтервалів) → `wilcox.test(headway ~ peak)` (Mann–Whitney). У Methodology 2 речення, чому обрано тест.
- **H2 (слоти / типи):** `kruskal.test(sched_headway_min ~ time_slot)` і те саме по `transport_type`. Значущо → `pairwise.wilcox.test(..., p.adjust.method = "holm")`, щоб побачити, які саме пари відрізняються.
- **H3 (незалежність):** створити `freq_cat = cut(sched_headway_min, breaks = c(0, 10, 20, Inf), labels = c("частий","середній","рідкісний"))` на пікових слотах → `table(transport_type, freq_cat)` → `chisq.test()`. Перевірити очікувані частоти (`$expected` ≥ 5); якщо менші — об'єднати категорії або поправка Єйтса.
- **H4 (райони):** `chisq.test(x = capacity_по_районах, p = частка_населення_району)`. Стандартизовані залишки (`$stdres`) покажуть, які райони недо-/переобслужені — це прямий матеріал для карти.
- **H5 (експоненційність):** оцінити λ = 1/mean(headway) для одного слота одного типу транспорту → `ks.test(headway, "pexp", rate = lambda)` + графік ECDF проти теоретичної CDF. Чесно зазначити: λ оцінена з тих самих даних, тому p-value наближене — і це гарна ремарка в Discussion.
- **Бонус (bootstrap):** 5000 ресемплів різниці середніх пік vs непік → percentile-CI; порівняти висновок із H1.

**Готово, коли:** для кожної гіпотези є: діагностика → обраний тест → p-value → висновок людською мовою.
