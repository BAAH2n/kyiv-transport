## 1. Для Methodology

**Дані:** Портал даних Києва, КП «Київпастранс» (Open Data Licence), завантажено 11.07.2026. Прямі лінки — PLAN.md §2.

**Дизайн:** плановий розклад буднього дня (service_id з monday = 1). Headway — на першій контрольній точці (stop_sequence == 1) = відправлення з кінцевої; фільтр 0 < headway < 120 хв. Слоти: ранковий пік 7–10, день 10–17, вечірній пік 17–20, вечір/ніч.

**Типи транспорту (route_type у файлі, стандарт GTFS):** 0 = трамвай (17 маршрутів), 3 = автобус (99), 11 = тролейбус (44). Разом 160. Зручний патерн: префікс route_id = 1_/2_/3_ відповідає трамвай/тролейбус/автобус.

## 2. Для Discussion (limitations)

- Трамвай 12 (1_37): у напрямку 1 розраховані інтервали (~22–24 хв) перевищують заявлені 11–17 хв; асиметрія кількості відправлень між напрямками (7–8 vs 11–25 на слот). Ймовірна причина — неповнота рейсів у GTFS-фіді (проблема відсутніх рейсів, зокрема депо-рейсів, підтверджена опрацьованими зверненнями громадян на сторінці набору).
- Аналізуємо планову пропозицію, не фактичний рух (частково закриває бонус-пілот «план vs факт»).
- GTFS — лише Київпастранс; маршрутки без розкладу (порівняння перевізників — лише за парком/місткістю).
- stop_times — лише контрольні точки.
- Населення району — проксі попиту (валідації е-квитка не публікуються).
- (Якщо буде бонус) GPS-пілот: один вихідний день, шум, неповне покриття.
